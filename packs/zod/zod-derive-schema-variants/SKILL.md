# zod-derive-schema-variants

## Overview
Keep schema variants **DRY** by deriving them from a base Zod schema.

This skill focuses on:
- **derivation** (`.partial()`, `.pick()`, `.omit()`)
- **extension/composition** (`.extend()`, composing schemas as fields)

## Procedures
Step 1: Select the minimal pattern set (routing)
0) Read `references/mental-model-sets.md`.
1) Read `references/catalog.md`.
2) Pick 1–2 relevant patterns (`P-..`) and any matching pitfalls (`X-..`).
3) If out of scope, stop and route to:
   - API JSON response boundaries → `zod-validate-api-boundaries`
   - input coercion/rehydration → `zod-normalize-inputs`

Step 2: Derive variants (progressive disclosure)
1) Read only the selected sections in `references/patterns.md` (and `references/pitfalls.md` if needed).
2) Ensure there is a **base schema** that represents the canonical shape (single source of truth).
3) Derive the smallest variant that matches the use case:
   - PATCH/update input → `.partial()` (+ `.omit(...)` for immutable keys)
   - public view → `.omit(...)` sensitive fields
   - subset payload → `.pick(...)`
   - new variant with extra fields → `.extend(...)`
4) Derive TypeScript types via `z.infer` from the variant (do not duplicate interfaces).

Step 3: Produce the required output
1) Add/update base schema and derived variants.
2) Update the call site(s) to use the appropriate variant.
3) Add a short rationale (2–5 bullets). Optional: `Refs used: P-.., X-..`.

Step 4: Validate and close
1) If repo instructions define a gate, run it; otherwise typecheck the affected module(s).
2) If issues appear, fix the **declaration** (base schema or variant) rather than adding casts.

## Error handling
- **No base schema exists:** create the smallest base schema first, then derive variants.
- **Variant unclear (create vs update vs view):** stop and ask which fields are mutable, required, and/or sensitive.
- **Drift detected:** replace copy/paste schemas with derivations (see `X-01`).
